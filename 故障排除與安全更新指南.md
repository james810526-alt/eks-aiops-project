## 🛠️ 第一部分：建置故障排除紀錄 (Troubleshooting)

### 1. Stack 08 Lambda `AccessDeniedException` 故障
* **異常現象**：
  在工程師收到告警郵件並點選 **【同意一鍵修復】** 或 **【拒絕執行】** 時，網頁彈出 `Internal Error`，查看 CloudWatch Logs 顯示 `AccessDeniedException`，指出 Lambda 無法對 DynamoDB 執行 `UpdateItem`。
* **根本原因**：
  原本 Lambda 執行角色的 IAM 權限僅給予 `dynamodb:GetItem` 與 `dynamodb:PutItem`，但 Lambda 代碼在 `/approve` 與 `/reject` 路由中，需要透過 `update_item` 將快取表中的警報狀態從 `PENDING` 更新為 `APPROVED` 或 `REJECTED`。
* **排除方案**（已更新至 [nkc201-17-08-aiops-stack.yaml](file:///c:/Users/USER/Desktop/專題資料/CloudFromation/nkc201-17-08-aiops-stack.yaml)）：
  在 Lambda 權限中補上 `dynamodb:UpdateItem`：
  ```yaml
  - Effect: Allow
    Action:
      - dynamodb:GetItem
      - dynamodb:PutItem
      - dynamodb:UpdateItem
    Resource: !GetAtt AlertCacheTable.Arn
  ```

### 2. CodeBuild 於私有子網 VPC 啟動失敗
* **異常現象**：
  AIOps 修復階段啟動時，CodeBuild 專案無法成功與 VPC 連線，建置狀態一直卡在 `SUBMITTED` 或 `PROVISIONING`，最後回報網路超時失敗。
* **根本原因**：
  CodeBuild 被配置在私有子網（Private Subnet）中執行以存取私有 EKS 叢集。AWS 官方文件指出，當 CodeBuild 使用 VPC 連線時，其 Service Role 必須擁有在 VPC 內**建立、查詢與刪除 Elastic Network Interface (ENI) 網路介面**的 EC2 權限，而原本的 IAM 角色缺少了這部分。
* **排除方案**（已更新至 [nkc201-17-03-iam-stack.yaml](file:///c:/Users/USER/Desktop/專題資料/CloudFromation/nkc201-17-03-iam-stack.yaml)）：
  在 `CodeBuildRole` 中補上 EC2 建立與管理 ENI 的權限：
  ```yaml
  - Effect: Allow
    Action:
      - ec2:CreateNetworkInterface
      - ec2:DescribeDhcpOptions
      - ec2:DescribeNetworkInterfaces
      - ec2:DeleteNetworkInterface
      - ec2:DescribeSubnets
      - ec2:DescribeSecurityGroups
      - ec2:DescribeVpcs
    Resource: "*"
  - Effect: Allow
    Action:
      - ec2:CreateNetworkInterfacePermission
    Resource: !Sub "arn:aws:ec2:${AWS::Region}:${AWS::AccountId}:network-interface/*"
    Condition:
      StringEquals:
        ec2:AuthorizedService: codebuild.amazonaws.com
  ```

### 3. SSM Bastion Host 鏡像 ID 動態語法錯誤
* **異常現象**：
  部署 Stack 05 時，CloudFormation 回報 `ImageId` 解析錯誤而導致 Rollback。
* **根本原因**：
  原本範本中使用 Python 自動填入固定的 `ami-xxxx`，但在孟買區該 AMI ID 可能過期。隨後改用 AWS SSM 參數存儲動態參考最新 AL2023 時，動態語法格式不符合 CloudFormation 規範。
* **排除方案**（已更新至 [nkc201-17-05-nodegroup-stack.yaml](file:///c:/Users/USER/Desktop/專題資料/CloudFromation/nkc201-17-05-nodegroup-stack.yaml)）：
  將鏡像動態參考修正為標準的 SSM Resolve 雙大括號格式：
  ```yaml
  ImageId: "{{resolve:ssm:/aws/service/ami-amazon-linux-latest/al2023-ami-kernel-6.1-x86_64}}"
  ```

### 4. Kubernetes Ingress ALB 無法建立（未標記 Subnet）
* **異常現象**：
  Ingress 部署成功但一直無法取得負載平衡器 DNS 地址（Address 欄位為空）。
* **根本原因**：
  AWS Load Balancer Controller 需要透過 Subnet 的 Tags 來自動搜尋對應的子網來部署負載平衡器。
* **排除方案**（已更新至 [nkc201-17-01-network-stack.yaml](file:///c:/Users/USER/Desktop/專題資料/CloudFromation/nkc201-17-01-network-stack.yaml)）：
  確保 Public Subnet 打上：`kubernetes.io/role/elb = 1`；Private Subnet 打上：`kubernetes.io/role/internal-elb = 1`。

### 5. Managed Node Group 未掛上自訂 Node SG
* **異常現象**：
  Node Group 與 Pods 都正常，但 Pod 連線 RDS port 3306 被拒絕；檢查 Worker EC2 時找不到 Stack 02 建立的 Node SG。
* **根本原因**：
  `AWS::EKS::Nodegroup` 原本沒有使用 Launch Template，因此自訂 Node SG 並未掛到 Worker Nodes；RDS SG 卻以該 Node SG 作為唯一來源。
* **排除方案**（已更新至 `CloudFormation/nkc201-17-05-nodegroup-stack.yaml`）：
  建立 `AWS::EC2::LaunchTemplate`，同時掛上自訂 Node SG 與 Stack 04 輸出的 EKS Cluster SG，並把 20 GiB gp3 root volume 移入 Launch Template。已存在且原本未使用 Launch Template 的 Node Group 應建立新 Node Group 遷移，不要直接原地切換。

### 6. ALB Controller IAM Policy 版本漂移與 ALB SG 未使用
* **異常現象**：
  IAM Stack 顯示 `CREATE_COMPLETE`，但 Controller 建立 ALB 時出現 `AccessDenied`，或 Stack 02 的 ALB SG 完全沒有掛到實際 Load Balancer。
* **根本原因**：
  舊 inline policy 缺少目前 Controller 所需的 tagging、listener 等權限；Ingress 也沒有指定既有 ALB SG。
* **排除方案**：
  Stack 03 已固定 AWS Load Balancer Controller `v3.4.0` 官方 managed policy，Helm chart 同步固定 `3.4.0`。`Kubernetes/web-prod-app.yaml` 使用 SG placeholder，透過 `Kubernetes/deploy-web-prod.sh` 注入 Stack 02 的 `AlbSecurityGroupId` 後才套用。

## 🔒 第二部分：資安討論與 DevSecOps 安全防護更新

我們針對整個 AIOps 專題架構，實施了多項超越一般專題（達企業生產級標準）的安全升級：

### 🚨 安全漏洞探討 1：遠端命令注入風險 (eval Remote Code Execution)
* **安全隱患**：
  原本 CodeBuild 在執行修復時，直接使用 `eval "$REM_COMMAND"` 執行由 Amazon Bedrock AI 自動生成的 shell 指令。即使有人工審查，由於 eval 會解析 `;`、`&&`、`|`、反引號等複合字元，一旦 Bedrock 被惡意注入（Prompt Injection），可能生成如 `rm -rf` 或 `aws iam` 等毀滅性命令。
* **DevSecOps 更新方案**：
  1. **限制 AI 只輸出結構化 JSON**：Bedrock 被嚴格限制僅能輸出 `ACTION_JSON`，不提供 raw shell 語句。
  2. **Lambda 檢驗與白名單過濾**：Lambda 對 JSON 進行正則表達式檢驗，僅允許 `web-prod` 與 `aiops` 命名空間，且僅限低風險的 `restart_deployment`/`scale_deployment`/`delete_pod` 動作，防範所有危險字元。
  3. **安全模板拼接**：由 Lambda 內部將安全值填入寫死的模板（不使用 `eval`），CodeBuild 接收後直接以字串執行，從根本杜絕命令注入漏洞。

---

### 🚨 安全漏洞探討 2：審批連結防爬蟲與狀態並發原子鎖 (GET-to-POST & Conditional Update)
* **安全隱患**：
  * **郵件預爬機制（Prefetch / Crawl）**：企業郵件安全閘道或郵件用戶端常會對信件中的 GET 連結進行預爬。原設計直接將 `/approve` 與 `/reject` 設為 GET 操作，導致工程師還沒點擊郵件，自動修復就已經被預先爬蟲觸發核准。
  * **雙擊與重放攻擊**：審批連結僅透過 `hash` 參數驗證，且未限制狀態移轉，可能遭工程師雙擊或惡意者重放呼叫，造成重複觸發 CodeBuild 引起叢集震盪。
* **DevSecOps 更新方案**（已更新至 [nkc201-17-08-aiops-stack.yaml](file:///c:/Users/USER/Desktop/專題資料/CloudFromation/nkc201-17-08-aiops-stack.yaml)）：
  1. **UUID 一次性 Token**：生成隨機 UUID `OneTimeToken` 寫入 DynamoDB 並夾帶於 URL 中作為一次性存取權杖。
  2. **郵件預爬防護（GET-to-POST 雙向路由確認）**：將 API Gateway 路由升級為 `ANY` 模式。當工程師點擊郵件時以 `GET` 請求連線，Lambda 會回傳一個使用 HSL 設計的精美 HTML 二次確認頁卡片（使用 Inter 字體與毛玻璃特效）。爬蟲僅會預爬到該卡片網頁，而卡片中的確認按鈕改用 **HTML Form POST** 表單提交。只有當工程師親自點擊按鈕發送 `POST` 請求時，Lambda 才會真正更新狀態並觸發自動修復，徹底阻絕爬蟲誤觸。
  3. **DynamoDB 狀態原子鎖**：在進行狀態轉移至 `APPROVED` 或 `REJECTED` 時，Lambda 會透過 DynamoDB 的 `ConditionExpression="Status = :pending"` 表達式進行原子條件更新。一旦有搶鎖成功的請求更新了狀態，任何重複或雙擊的請求皆會因為條件不符而直接被丟棄。

---

### 🚨 安全漏洞探討 3：告警風暴並發與 race condition
* **安全隱患**：
  在 K8s 發生 Pod 頻繁重啟時，Webhook 可能在極短時間內（以毫秒計）發送多筆相同警報。由於 Bedrock 的分析需要 2-3 秒，在第一筆警報尚未寫入 DynamoDB 之前，後續的警報便會通過 check，導致重複寄發多封郵件與浪費 API 額度。
* **DevSecOps 更新方案**：
  1. **原子寫入鎖 (Atomic Lock)**：Lambda 接收到 webhook 時，直接在第一時間使用 DynamoDB **條件寫入**（`ConditionExpression`）進行鎖定（設定狀態為 `ANALYZING`）。只有搶鎖成功的請求才能繼續調用 Bedrock，其餘並發警報會直接被丟棄。

---

### 🚨 安全漏洞探討 4：跳板機權限過大與網路暴露
* **安全隱患**：
  原本跳板機 EC2 使用了與 EKS Worker Node 相同的 IAM Role（權限過大），且若開放 SSH 22 端口，會面臨暴力破解與網路掃描風險。
* **DevSecOps 更新方案**：
  1. **獨立 BastionRole**：建立專屬 `BastionRole`，僅給予最小的 `SSMManagedInstanceCore` 與 Assume `EngineerRole` 的權限。
  2. **無公網 IP、關閉 Port 22**：跳板機部署於私有子網，無公網 IP，不開 22 端口。工程師完全透過 **AWS Systems Manager Session Manager** 安全登入跳板機，實施零信任安全。

---

### 🚨 安全漏洞探討 5：CodeBuild 供應鏈劫持與內網邊界
* **安全隱患**：
  CodeBuild 原本每次都從公網下載 `latest` kubectl，且未驗證 checksum，有遭受供應鏈投毒的風險。
* **DevSecOps 更新方案**（已更新至 [nkc201-17-08-aiops-stack.yaml](file:///c:/Users/USER/Desktop/專題資料/CloudFromation/nkc201-17-08-aiops-stack.yaml)）：
  1. **版本與雜湊鎖定**：CodeBuild 的 kubectl 被鎖定為 `v1.34.0`，並在下載時拉取官方的 `.sha256` 校驗碼進行 **SHA256 強制雜湊校驗**。
  2. **AWS PrivateLink 全內網閉環**：Lambda 與 Bedrock、STS 的連線部署了專屬的 VPC 端點 (VPC Endpoints)，使整個 AI 維運流程全部運作於 VPC 封閉的內網中，防止流量暴露於 Internet。

---

### 🚨 安全漏洞探討 6：Webhook 未授權存取與費用濫用風險 (Token Authentication)
* **安全隱患**：
  原本對外開放的 `/webhook` 端點缺乏認證機制。惡意攻擊者若刺探到 API Gateway Webhook 端點，即可發送大量偽造或惡意的告警 JSON 封包，導致 Lambda 頻繁調用 Amazon Bedrock 進行 AI 診斷，進而耗盡 AWS API 額度並產生高額的 Bedrock 帳單（即 DDoS 騷擾攻擊與費用攻擊）。
* **DevSecOps 更新方案**（已更新至 [nkc201-17-08-aiops-stack.yaml](file:///c:/Users/USER/Desktop/專題資料/CloudFromation/nkc201-17-08-aiops-stack.yaml) 及 [k8sgpt-operator-config.yaml](file:///c:/Users/USER/Desktop/專題資料/Kubernetes/k8sgpt-operator-config.yaml)）：
  1. **Webhook 查詢 Token 認證**：在 Lambda Webhook 路由處理中，引入了祕鑰校驗邏輯（要求查詢參數中必須帶有 `token=eks-aiops-webhook-secret-token`）。只有校驗通過的請求才會執行解密診斷，防範未經授權的虛假告警。
  2. **K8sGPT Webhook 配置同步**：在 Kubernetes 中部署的 K8sGPT 掃描器，其 `Secret` 內的 webhook url 已同步變更並附加該 token，實現可信來源的 Webhook 對接。
  3. **Lambda 預留並發限制 (Reserved Concurrency)**：在 Lambda 屬性中配置了 `ReservedConcurrentExecutions: 2`。即使遇到告警風暴或大量 Webhook 並發，Lambda 的最大執行數會被鎖定在 2，強制對過量請求進行 Throttling，作為防範 AWS Bedrock 費用溢出的最後防波堤。

---

### 🚨 安全漏洞探討 7：VPC Endpoints 安全群組（SG）來源過寬漏洞
* **安全隱患**：
  原設計中，Stack 08 建立的 3 個 VPC Endpoint（Bedrock Runtime, STS, Logs）安全群組在 Ingress 中將 443 埠口開放給了 `0.0.0.0/0`。雖然這些 Interface 端點僅具備 VPC 內部的私有 IP 且僅能在 VPC 內連通，但對外放行 `0.0.0.0/0` 不符合最小權限安全原則（Least Privilege Principle）。
* **DevSecOps 更新方案**（已更新至 [nkc201-17-08-aiops-stack.yaml](file:///c:/Users/USER/Desktop/專題資料/CloudFromation/nkc201-17-08-aiops-stack.yaml)）：
  1. **收斂來源至 VpcCidr**：在範本中引進 `VpcCidr` 參數，將這 3 個 VPC Endpoint 的安全群組來源 IP 完全收緊至 VPC IP 網段 `10.20.0.0/16`。確保只有本 VPC 網路內部受信任的資源（如 Lambda 與工作節點）才能存取並解析此端點。

### 🚨 安全漏洞探討 8：Bedrock IAM 權限過寬與資源未鎖定風險
* **安全隱患**：
  原設計中，Lambda 呼叫 Bedrock 的 IAM Policy 採用資源萬用字元：`Resource: 'arn:aws:bedrock:*::foundation-model/*'`。這代表 Lambda 可以調用 AWS 任何區域的任何 AI 模型，一旦憑證外流，可能被利用來呼叫昂貴的大模型，造成帳單失控。
* **DevSecOps 更新方案**（已更新至 [nkc201-17-08-aiops-stack.yaml](file:///c:/Users/USER/Desktop/專題資料/CloudFromation/nkc201-17-08-aiops-stack.yaml)）：
  1. **資源 ARN 精確鎖定**：將 IAM Policy 中的資源範圍限制為僅能使用當前區域（ap-south-1）的 Claude 3 Haiku 模型（`arn:aws:bedrock:${AWS::Region}::foundation-model/anthropic.claude-3-haiku-*-v1:0`），遵循最小特權原則，鎖定 AI 邊界。

## 📐 第三部分：架構設計權衡與抉擇 (Architecture Design Decisions)

在 AWS 雲端基礎設施部署設計中，我們對「跨 Stack 資源對接機制」進行了深入的架構權衡，並決定採取特定的防護機制：

### 1. 跨 Stacks 對接：手動傳參（Parameter-Passing） vs. 自動導出（Export/ImportValue）
* **現狀設計**：
  在 Stack 01（網路層）的 Outputs 中，我們刻意**不對** `VpcId` 或 `SubnetIds` 進行 `Export` 宣告。在後續的 Stack 04、05、06、08 中，採用「手動傳入參數（Parameters）」的方式接收網路層 ID，而非使用 `Fn::ImportValue` 自動引進。
* **權衡考量與決策原因**：
  * **防止「巢狀依賴鎖死」（Dependency Locking）**：
    AWS CloudFormation 規定：*只要有任何後面的 Stack 正在引用某個 Export 的資源，前面的父 Stack 就絕對不允許被刪除或修改。*
    在專題測試與開發階段，學員需要頻繁地刪除、重建或修改資源（例如修改子網路規劃、重建網路層以重新獲得乾淨的環境）。如果使用了 `ImportValue` 強耦合，當要刪除基礎設施時，會被強制卡死，必須嚴格按照 `08 ➔ 07 ➔ 06 ➔ 05 ➔ 04 ➔ 03 ➔ 02 ➔ 01` 的反向順序一個一個手動刪除。一旦中間有任何一個 Stack 刪除卡住，整個網路層就會被完全鎖死，極易造成雲端帳單失控或除錯困難。
  * **架構決策**：
    我們選擇**「維持手動傳參現狀」**。雖然在主控台部署時需要手動複製貼上 VPC 與 Subnet ID，步驟稍微繁瑣，但極大地釋放了 Stack 之間的耦合度，讓專題專案在清理資源、重建測試環境時具備最高的靈活性與容錯率。

### 2. 未來架構優化與安全加固建議 (Future Security & Architecture Enhancements)
針對本專題的基礎設施與網路設定，我們進行了全面的 DevSecOps 安全防護評估，並將以下 5 個優化項目作為未來生產環境部署的加固方向：

* **① ALB 至 Node 節點安全群組（SG）收斂（已完成加固）**
  * **現狀與安全防護**：原本 Stack 02 的 Node SG 針對 ALB SG 放行了全 TCP 埠口（0-65535），權限過寬。
  * **實施方案**（已更新至 [nkc201-17-02-security-stack.yaml](file:///c:/Users/USER/Desktop/專題資料/CloudFromation/nkc201-17-02-security-stack.yaml)）：
    已收斂此安全群組規則，拆分為 `AlbToNodeIngressNodePorts`（放行 EKS 標準 NodePort 範圍 `30000-32767`，以保留與 Instance 模式的相容性）與 `AlbToNodeIngressHttp`（專屬放行業務暨健康檢查的 Port `80`，配合 `target-type: ip` 模式），徹底消除節點敏感系統埠口暴露之隱患。
* **② VPC Endpoints 安全群組（SG）來源鎖定（已完成加固）**
  * **現狀與安全防護**：Stack 08 建立的 3 個 VPC 端點（Bedrock Runtime, STS, Logs）安全群組之 Ingress 來源原設定為 `0.0.0.0/0`，權限過寬。
  * **實施方案**：已在 [nkc201-17-08-aiops-stack.yaml](file:///c:/Users/USER/Desktop/專題資料/CloudFromation/nkc201-17-08-aiops-stack.yaml) 中完成加固，來源 IP 網段已收緊限制為 `VpcCidr` (`10.20.0.0/16`)，實踐最嚴格的網路邊界存取控制。
* **③ 網路單點故障與高可用（Single NAT Gateway）**
  * **現狀與省錢取捨**：為節省專題測試費用，本架構採用了 Single NAT Gateway（部署於單一可用區）。若該可用區（AZ）發生故障，將導致所有私有子網路無法出網。
  * **優化建議**：在正式生產環境中，應切換為 Multi-AZ 模式（在 3 個可用區各部署一個 NAT Gateway），以達到真正的高可用與防單點故障能力。
* **④ 零外網環境下之 SSM 連線端點依賴**
  * **現狀與網路依賴**：目前跳板機（Bastion Host）是依賴 NAT Gateway 連上 AWS SSM 服務以進行 Session Manager 登入。如果未來關閉 NAT 走向「完全零外網」架構，跳板機將失效。
  * **優化建議**：未來若改為純 Endpoint 架構，必須補上 `ssm`、`ssmmessages` 與 `ec2messages` 三個 VPC Endpoint，確保在零外網環境下跳板機依然能安全管理 EKS。
* **⑤ 開啟 VPC Flow Logs 網路審計**
  * **現狀與審計規劃**：目前未開啟 VPC Flow Logs。
  * **優化建議**：未來應開啟 VPC Flow Logs 並將網路流量日誌傳送至 CloudWatch Logs 進行審計分析，提升整體網路層的可觀測性（Observability）與 DevSecOps 防護力。
