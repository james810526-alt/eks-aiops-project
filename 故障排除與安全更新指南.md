# EKS AIOps 專題 - 故障排除與安全更新指南 (Troubleshooting & Security Updates)

本指南統整了本專題自 **Stack 01 至 Stack 08** 在建置過程中遭遇的關鍵故障錯誤、安全漏洞探討，以及我們對應實施的 **DevSecOps 生產級安全加固措施**。

---

## 🛠️ 第一部分：建置故障排除紀錄 (Troubleshooting)

### 1. Stack 08 Lambda `AccessDeniedException` 故障
* **異常現象**：
  在工程師收到告警郵件並點選 **【同意一鍵修復】** 或 **【拒絕執行】** 時，網頁彈出 `Internal Error`，查看 CloudWatch Logs 顯示 `AccessDeniedException`，指出 Lambda 無法對 DynamoDB 執行 `UpdateItem`。
* **根本原因**：
  原本 Lambda 執行角色的 IAM 權限僅給予 `dynamodb:GetItem` 與 `dynamodb:PutItem`，但 Lambda 代碼在 `/approve` 與 `/reject` 路由中，需要透過 `update_item` 將快取表中的警報狀態從 `PENDING` 更新為 `APPROVED` 或 `REJECTED`。
* **排除方案**（已更新至 [nkc201-17-08-aiops-stack.yaml](file:///c:/Users/USER/Desktop/專題資料/CloudFromation/nkc201-17-08-aiops-stack.yaml)）：
  在 Lambda 權限中補上 `dynamodb:UpdateItem`：
  ```yaml
  - Effect: Allow
    Action:
      - dynamodb:GetItem
      - dynamodb:PutItem
      - dynamodb:UpdateItem
    Resource: !GetAtt AlertCacheTable.Arn
  ```

### 2. CodeBuild 於私有子網 VPC 啟動失敗
* **異常現象**：
  AIOps 修復階段啟動時，CodeBuild 專案無法成功與 VPC 連線，建置狀態一直卡在 `SUBMITTED` 或 `PROVISIONING`，最後回報網路超時失敗。
* **根本原因**：
  CodeBuild 被配置在私有子網（Private Subnet）中執行以存取私有 EKS 叢集。AWS 官方文件指出，當 CodeBuild 使用 VPC 連線時，其 Service Role 必須擁有在 VPC 內**建立、查詢與刪除 Elastic Network Interface (ENI) 網路介面**的 EC2 權限，而原本的 IAM 角色缺少了這部分。
* **排除方案**（已更新至 [nkc201-17-03-iam-stack.yaml](file:///c:/Users/USER/Desktop/專題資料/CloudFromation/nkc201-17-03-iam-stack.yaml)）：
  在 `CodeBuildRole` 中補上 EC2 建立與管理 ENI 的權限：
  ```yaml
  - Effect: Allow
    Action:
      - ec2:CreateNetworkInterface
      - ec2:DescribeDhcpOptions
      - ec2:DescribeNetworkInterfaces
      - ec2:DeleteNetworkInterface
      - ec2:DescribeSubnets
      - ec2:DescribeSecurityGroups
      - ec2:DescribeVpcs
    Resource: "*"
  - Effect: Allow
    Action:
      - ec2:CreateNetworkInterfacePermission
    Resource: !Sub "arn:aws:ec2:${AWS::Region}:${AWS::AccountId}:network-interface/*"
    Condition:
      StringEquals:
        ec2:AuthorizedService: codebuild.amazonaws.com
  ```

### 3. SSM Bastion Host 鏡像 ID 動態語法錯誤
* **異常現象**：
  部署 Stack 05 時，CloudFormation 回報 `ImageId` 解析錯誤而導致 Rollback。
* **根本原因**：
  原本範本中使用 Python 自動填入固定的 `ami-xxxx`，但在孟買區該 AMI ID 可能過期。隨後改用 AWS SSM 參數存儲動態參考最新 AL2023 時，動態語法格式不符合 CloudFormation 規範。
* **排除方案**（已更新至 [nkc201-17-05-nodegroup-stack.yaml](file:///c:/Users/USER/Desktop/專題資料/CloudFromation/nkc201-17-05-nodegroup-stack.yaml)）：
  將鏡像動態參考修正為標準的 SSM Resolve 雙大括號格式：
  ```yaml
  ImageId: "{{resolve:ssm:/aws/service/ami-amazon-linux-latest/al2023-ami-kernel-6.1-x86_64}}"
  ```

---

### 4. Kubernetes Ingress ALB 無法建立（未標記 Subnet）
* **異常現象**：
  Ingress 部署成功但一直無法取得負載平衡器 DNS 地址（Address 欄位為空）。
* **根本原因**：
  AWS Load Balancer Controller 需要透過 Subnet 的 Tags 來自動搜尋對應的子網來部署負載平衡器。
* **排除方案**（已更新至 [nkc201-17-01-network-stack.yaml](file:///c:/Users/USER/Desktop/專題資料/CloudFromation/nkc201-17-01-network-stack.yaml)）：
  確保 Public Subnet 打上：`kubernetes.io/role/elb = 1`；Private Subnet 打上：`kubernetes.io/role/internal-elb = 1`。

## 🔒 第二部分：資安討論與 DevSecOps 安全防護更新

我們針對整個 AIOps 專題架構，實施了多項超越一般專題（達企業生產級標準）的安全升級：

### 🚨 安全漏洞探討 1：遠端命令注入風險 (eval Remote Code Execution)
* **安全隱患**：
  原本 CodeBuild 在執行修復時，直接使用 `eval "$REM_COMMAND"` 執行由 Amazon Bedrock AI 自動生成的 shell 指令。即使有人工審查，由於 eval 會解析 `;`、`&&`、`|`、反引號等複合字元，一旦 Bedrock 被惡意注入（Prompt Injection），可能生成如 `rm -rf` 或 `aws iam` 等毀滅性命令。
* **DevSecOps 更新方案**：
  1. **限制 AI 只輸出結構化 JSON**：Bedrock 被嚴格限制僅能輸出 `ACTION_JSON`，不提供 raw shell 語句。
  2. **Lambda 檢驗與白名單過濾**：Lambda 對 JSON 進行正則表達式檢驗，僅允許 `web-prod` 與 `aiops` 命名空間，且僅限低風險的 `restart_deployment`/`scale_deployment`/`delete_pod` 動作，防範所有危險字元。
  3. **安全模板拼接**：由 Lambda 內部將安全值填入寫死的模板（不使用 `eval`），CodeBuild 接收後直接以字串執行，從根本杜絕命令注入漏洞。

---

### 🚨 安全漏洞探討 2：審批連結預測與二次點擊 (Token & State Transition)
* **安全隱患**：
  原設計中，審批連結僅透過 `hash` 參數（基於錯誤訊息雜湊，容易被預測）進行驗證，且 Lambda 沒有檢查狀態轉移。一旦工程師雙擊（Double-click）或連結洩漏，CodeBuild 會重複被觸發，造成叢集資源震盪與重放攻擊。
* **DevSecOps 更新方案**：
  1. **UUID 一次性 Token**：生成隨機的 UUID `OneTimeToken` 寫入 DynamoDB 並夾帶於 URL 中，大幅提升安全性。
  2. **PENDING 狀態校驗**：Lambda 在處理審批時，會嚴格要求當前狀態必須是 `PENDING`，驗證成功後**立即**更新為 `APPROVED` 才會觸發 CodeBuild，第二次點擊會因狀態已變更而直接被攔截拒絕。

---

### 🚨 安全漏洞探討 3：告警風暴並發與 race condition
* **安全隱患**：
  在 K8s 發生 Pod 頻繁重啟時，Webhook 可能在極短時間內（以毫秒計）發送多筆相同警報。由於 Bedrock 的分析需要 2-3 秒，在第一筆警報尚未寫入 DynamoDB 之前，後續的警報便會通過 check，導致重複寄發多封郵件與浪費 API 額度。
* **DevSecOps 更新方案**：
  1. **原子寫入鎖 (Atomic Lock)**：Lambda 接收到 webhook 時，直接在第一時間使用 DynamoDB **條件寫入**（`ConditionExpression`）進行鎖定（設定狀態為 `ANALYZING`）。只有搶鎖成功的請求才能繼續調用 Bedrock，其餘並發警報會直接被丟棄。

---

### 🚨 安全漏洞探討 4：跳板機權限過大與網路暴露
* **安全隱患**：
  原本跳板機 EC2 使用了與 EKS Worker Node 相同的 IAM Role（權限過大），且若開放 SSH 22 端口，會面臨暴力破解與網路掃描風險。
* **DevSecOps 更新方案**：
  1. **獨立 BastionRole**：建立專屬 `BastionRole`，僅給予最小的 `SSMManagedInstanceCore` 與 Assume `EngineerRole` 的權限。
  2. **無公網 IP、關閉 Port 22**：跳板機部署於私有子網，無公網 IP，不開 22 端口。工程師完全透過 **AWS Systems Manager Session Manager** 安全登入跳板機，實施零信任安全。

---

### 🚨 安全漏洞探討 5：CodeBuild 供應鏈劫持與內網邊界
* **安全隱患**：
  CodeBuild 原本每次都從公網下載 `latest` kubectl，且未驗證 checksum，有遭受供應鏈投毒的風險。
* **DevSecOps 更新方案**：
  1. **版本與雜湊鎖定**：CodeBuild 的 kubectl 被鎖定為 `v1.34.0`，並在下載時拉取官方的 `.sha256` 校驗碼進行 **SHA256 強制雜湊校驗**。
  2. **AWS PrivateLink 全內網閉環 (P2)**：Lambda 與 Bedrock、STS 的連線部署了專屬的 VPC 端點 (VPC Endpoints)，使整個 AI 維運流程全部運作於 VPC 封閉的內網中，防止流量暴露於 Internet。

## 📐 第三部分：架構設計權衡與抉擇 (Architecture Design Decisions)

在 AWS 雲端基礎設施部署設計中，我們對「跨 Stack 資源對接機制」進行了深入的架構權衡，並決定採取特定的防護機制：

### 1. 跨 Stacks 對接：手動傳參（Parameter-Passing） vs. 自動導出（Export/ImportValue）
* **現狀設計**：
  在 Stack 01（網路層）的 Outputs 中，我們刻意**不對** `VpcId` 或 `SubnetIds` 進行 `Export` 宣告。在後續的 Stack 04、05、06、08 中，採用「手動傳入參數（Parameters）」的方式接收網路層 ID，而非使用 `Fn::ImportValue` 自動引進。
* **權衡考量與決策原因**：
  * **防止「巢狀依賴鎖死」（Dependency Locking）**：
    AWS CloudFormation 規定：*只要有任何後面的 Stack 正在引用某個 Export 的資源，前面的父 Stack 就絕對不允許被刪除或修改。*
    在專題測試與開發階段，學員需要頻繁地刪除、重建或修改資源（例如修改子網路規劃、重建網路層以重新獲得乾淨的環境）。如果使用了 `ImportValue` 強耦合，當要刪除基礎設施時，會被強制卡死，必須嚴格按照 `08 ➔ 07 ➔ 06 ➔ 05 ➔ 04 ➔ 03 ➔ 02 ➔ 01` 的反向順序一個一個手動刪除。一旦中間有任何一個 Stack 刪除卡住，整個網路層就會被完全鎖死，極易造成雲端帳單失控或除錯困難。
  * **架構決策**：
    我們選擇**「維持手動傳參現狀」**。雖然在主控台部署時需要手動複製貼上 VPC 與 Subnet ID，步驟稍微繁瑣，但極大地釋放了 Stack 之間的耦合度，讓專題專案在清理資源、重建測試環境時具備最高的靈活性與容錯率。
