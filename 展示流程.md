# EKS AIOps 智能維運專題 - 展示與 Live Demo 指南

本指南設計了一套針對評審、面試官或專題發表的 **Live Demo 故事線與操作流程**。透過此流程，您可以完整展現「故障發生 ➔ AI 智能診斷 ➔ 人工審批 ➔ 內網自動修復」的安全閉環。

---

## 🎭 故事線背景 (Scenario)
> **情境設定**：
> 「我們維運著一個高可用的 EKS 電商網站。晚上 20:00 限時促銷活動開跑，流量瞬間暴增。此時，剛入職的工程師在更新網頁服務時，不小心輸入了錯誤的鏡像檔版本（Image Tag），導致前台網站崩潰，用戶無法開啟網頁。」

---

## 🛠️ Live Demo 操作步驟

### 步驟一：展示正常狀態與注入故障

1. **展示正常運作的前台網頁**：
   在瀏覽器中開啟您的 ALB DNS 網址，向評審展示 Nginx 首頁能正常顯示。
2. **在 WSL 終端機中，故意注入故障**（將網頁 Image 改為不存在的 `nginx:1.999`）：
   ```bash
   kubectl set image deployment/web-demo web-demo=nginx:1.999 -n web-prod
   ```
3. **展示 Pod 異常狀態**：
   ```bash
   kubectl get pods -n web-prod
   ```
   * **畫面呈現**：Pod 狀態會轉為 `ImagePullBackOff` 或 `ErrImagePull`，此時重新整理瀏覽器網頁會無法顯示（網站掛掉）。

---

### 步驟二：AIOps 偵測與 AI 專家診斷

1. **說明 K8sGPT 與 Lambda 聯動**：
   * EKS 內部的 K8sGPT 掃描器自動偵測到 `web-demo` 的 Pod 出現 `ImagePullBackOff`。
   * K8sGPT 發送 Webhook 告警至 API Gateway `/webhook` 端點，觸發 AIOps Lambda。
2. **展示「防告警風暴」快取表**：
   * 開啟 AWS 控制台，進入 **DynamoDB**，展示 `eks-aiops-demo-alert-cache` 表。
   * **技術點說明**：
     > 「如果 Pod 不斷崩潰重啟，傳統監控會發送成百上千封垃圾郵件（告警風暴）。我們設計了 DynamoDB 作為快取，具備 **10分鐘 TTL 冷卻機制與原子寫入鎖**，當 Webhook 進來時會立即搶鎖，以防在 Bedrock 診斷期間（2-3秒）有重複告警並發進來而重複調用 LLM 與發送信件。」
3. **呼叫 Amazon Bedrock 專家診斷**：
   * Lambda 自動將錯誤日誌送交孟買區的 **Amazon Bedrock (Claude)**。
   * 展示 AWS Lambda 的 CloudWatch Logs，說明 AI 正在即時診斷該問題，並生成繁體中文診斷報告與結構化的 **Action JSON**。Lambda 會對其進行**嚴格的防注入檢驗與 Namespace/Action 白名單校驗**，再依據安全模板組裝成固定格式的指令，徹底消除 LLM 輸出任意 shell 指令的風險。

---

### 步驟三：工程師審查與一鍵核准 (Approve)

1. **開啟您的工程師信箱** (`james810526@gmail.com`)，展示收到的告警信：
   * 信中會有 Bedrock 自動生成的 **白話繁體中文診斷報告**（說明找不到鏡像，建議將其改回 `nginx:latest`）。
   * 信件最下方附帶兩個審批連結：
     * **👉【同意執行自動修復】** (`https://<apigw>/approve?hash=...&token=<OneTimeToken>`)
     * **👉【拒絕執行自動修復】** (`https://<apigw>/reject?hash=...&token=<OneTimeToken>`)
2. **執行人工審批 (Live Click)**：
   * 點選信中的 **【同意執行自動修復】**。
   * **技術點說明**：
     > 「為確保生產環境安全，我們絕對不讓 AI 直接修改叢集。透過 **Human-in-the-loop (人機協同)** 機制，修復指令必須經由工程師確認後才可啟動。此外，連結中帶有一次性的 **UUID 隨機 Token 認證**，Lambda 會嚴格校驗狀態必須為 `PENDING`，一經點擊立即變更狀態，**徹底防止重複點擊與越權執行**的漏洞。」
   * 瀏覽器會彈出 API Gateway 回傳的核准確認網頁，若再次點擊則會提示該動作已被執行而拒絕。

---

### 步驟四：VPC 內網安全自動修復與驗證

1. **展示 CodeBuild 私有建置**：
   * 打開 AWS **CodeBuild** 控制台，展示正在執行中的修復任務。
   * **技術點說明**：
     > 「因為我們的 EKS 控制面是完全私有化的（沒有 Public IP），且 Lambda 呼叫 Bedrock、STS 以及寫 Log 均透過 **AWS PrivateLink (VPC Endpoints)** 實現全內網通訊。隨後 Lambda 啟動了部署於 **VPC Private Subnet** 內的 CodeBuild 機器人，這台機器人會**固定下載 v1.34.0 kubectl 並進行官方 SHA256 強制校驗**以防止供應鏈劫持，最後在高度安全的內網環境中套用安全修復指令。」
2. **展示網站恢復正常**：
   * CodeBuild 執行 `kubectl set image deployment/web-demo web-demo=nginx:latest -n web-prod`。
   * 在 WSL 執行查詢：
     ```bash
     kubectl get pods -n web-prod
     ```
     * **畫面呈現**：Pod 狀態恢復為 `Running`。
   * 重新整理瀏覽器網頁，網站成功恢復連線！

## 🏆 專題架構五大技術亮點 (面試/發表必講)

1. **極致安全 (Zero Trust & Private-Only & PrivateLink)**：
   EKS API Server 關閉公網存取，工程師不開 Port 22 SSH，改用 **SSM Session Manager** 跳板機管理；Pod 權限採用最新的 **EKS Access Entry API** 進行細粒度控制。Lambda 與 Bedrock / STS / Logs 的通訊均走 **AWS PrivateLink (VPC Endpoints)** 專線，流量完全不經公網，實現全內網閉環。
2. **安全一次性 Token 與狀態機 (Human-in-the-loop Gatekeeper & One-Time Token)**：
   AI 只做分析與產生 Action JSON，最終修復指令必須經由工程師點選 Email 連結。連結附加隨機生成的一次性 **UUID Token**，Lambda 會嚴格驗證 Token 及 `PENDING` 狀態，點擊後狀態立即移轉，拒絕重複點擊或越權觸發，確保權限隔離與審查安全。
3. **防注入命令白名單安全模板 (Command Injection Mitigation)**：
   徹底移除 `eval "$REM_COMMAND"` 執行任意 shell 指令的隱患。Bedrock 僅能生成結構化 Action JSON，由 Lambda 解析並實施正規表達式校驗與危險字元攔截（防範 `;`、`&`、`|`、`` ` ``、`$` 等），限制僅可對特定 Namespace（`web-prod` 與 `aiops`）進行受控的 restart 或 scale 操作，從源頭扼殺命令注入漏洞。
4. **告警去重快取與原子鎖 (Alert Storm & Race Condition Mitigation)**：
   整合 DynamoDB 快取與 TTL 機制，同個資源的相同故障在冷卻期內只會通知一次。進一步引入 **Atomic Conditional Put 原子鎖** 機制，當警報觸發時先在資料庫鎖定該事件，防範 Bedrock 運算期間有並發的 duplicate webhooks 重複觸發 LLM 診斷與發送多封重複信件。
5. **CodeBuild 安全加固與版本鎖定 (CodeBuild Hardening & Checksum Lock)**：
   修復執行器（CodeBuild）將 kubectl 版本鎖定在與叢集一致的 `v1.34.0`，並在下載時自動請求官方 `.sha256` 校驗碼進行 **SHA256 強制雜湊校驗**，防止中間人攻擊或下載站點遭受污染（供應鏈劫持），安全層級提升至生產級標準。
