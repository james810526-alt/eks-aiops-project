# EKS AIOps 智能維運專題 - 展示與 Live Demo 指南

本指南設計了一套針對評審、面試官或專題發表的 **Live Demo 故事線與操作流程**。透過此流程，您可以完整展現「故障發生 ➔ AI 智能診斷 ➔ 人工審批 ➔ 內網自動修復」的安全閉環。

---

## 🎭 故事線背景 (Scenario)
> **情境設定**：
> 「我們維運著一個高可用的 EKS 電商網站。晚上 20:00 限時促銷活動開跑，流量瞬間暴增。此時，剛入職的工程師在更新網頁服務時，不小心輸入了錯誤的鏡像檔版本（Image Tag），導致前台網站崩潰，用戶無法開啟網頁。」

---

## 🛠️ Live Demo 操作步驟

### 步驟一：展示正常狀態與注入故障

1. **展示正常運作的前台網頁**：
   在瀏覽器中開啟您的 ALB DNS 網址，向評審展示 Nginx 首頁能正常顯示。
2. **在 WSL 終端機中，故意注入故障**（將網頁 Image 改為不存在的 `nginx:1.999`）：
   ```bash
   kubectl set image deployment/web-demo web-demo=nginx:1.999 -n web-prod
   ```
3. **展示 Pod 異常狀態**：
   ```bash
   kubectl get pods -n web-prod
   ```
   * **畫面呈現**：Pod 狀態會轉為 `ImagePullBackOff` 或 `ErrImagePull`，此時重新整理瀏覽器網頁會無法顯示（網站掛掉）。

---

### 步驟二：AIOps 偵測與 AI 專家診斷

1. **說明 K8sGPT 與 Lambda 聯動**：
   * EKS 內部的 K8sGPT 掃描器自動偵測到 `web-demo` 的 Pod 出現 `ImagePullBackOff`。
   * K8sGPT 發送 Webhook 告警至 API Gateway `/webhook` 端點，觸發 AIOps Lambda。
2. **展示「防告警風暴」快取表**：
   * 開啟 AWS 控制台，進入 **DynamoDB**，展示 `eks-aiops-demo-alert-cache` 表。
   * **技術點說明**：
     > 「如果 Pod 不斷崩潰重啟，傳統監控會發送成百上千封垃圾郵件（告警風暴）。我們設計了 DynamoDB 作為快取與 **10分鐘 TTL 冷卻機制**，在冷卻期內自動過濾重複告警，讓工程師能專注修復而非被信件淹沒。」
3. **呼叫 Amazon Bedrock 專家診斷**：
   * Lambda 自動將錯誤日誌送交孟買區的 **Amazon Bedrock (Claude)**。
   * 展示 AWS Lambda 的 CloudWatch Logs，說明 AI 正在即時診斷該問題並生成繁體中文的修復建議與 `kubectl` 單行指令。

---

### 步驟三：工程師審查與一鍵核准 (Approve)

1. **開啟您的工程師信箱** (`james810526@gmail.com`)，展示收到的告警信：
   * 信中會有 Bedrock 自動生成的 **白話繁體中文診斷報告**（說明找不到鏡像，建議將其改回 `nginx:latest`）。
   * 信件最下方附帶兩個審批連結：
     * **👉【同意執行自動修復】** (`https://<apigw>/approve?hash=...`)
     * **👉【拒絕執行自動修復】** (`https://<apigw>/reject?hash=...`)
2. **執行人工審批 (Live Click)**：
   * 點選信中的 **【同意執行自動修復】**。
   * **技術點說明**：
     > 「為確保生產環境安全，我們絕對不讓 AI 直接修改叢集。透過 **Human-in-the-loop (人機協同)** 機制，修復指令必須經由工程師確認後才可啟動。」
   * 瀏覽器會彈出 API Gateway 回傳的核准確認網頁。

---

### 步驟四：VPC 內網安全自動修復與驗證

1. **展示 CodeBuild 私有建置**：
   * 打開 AWS **CodeBuild** 控制台，展示正在執行中的修復任務。
   * **技術點說明**：
     > 「因為我們的 EKS 控制面是完全私有化的（沒有 Public IP），Lambda 無法直接存取。因此，Lambda 啟動了部署於 **VPC Private Subnet** 內的 CodeBuild 機器人。這台機器人使用與節點相同的安全群組（Security Group），在高度安全的內網環境中使用 `kubectl` 套用修復。」
2. **展示網站恢復正常**：
   * CodeBuild 執行 `kubectl set image deployment/web-demo web-demo=nginx:latest -n web-prod`。
   * 在 WSL 執行查詢：
     ```bash
     kubectl get pods -n web-prod
     ```
     * **畫面呈現**：Pod 狀態恢復為 `Running`。
   * 重新整理瀏覽器網頁，網站成功恢復連線！

---

## 🏆 專題架構四大技術亮點 (面試/發表必講)

1. **極致安全 (Zero Trust & Private-Only)**：
   EKS API Server 關閉公網存取，工程師不開 Port 22 SSH，改用 **SSM Session Manager** 跳板機管理；Pod 權限採用最新的 **EKS Access Entry API** 進行細粒度控制。
2. **人工審批閘口 (Human-in-the-loop Gatekeeper)**：
   AI 只做分析與生成指令，最終執行權依然保留在工程師點選 Email 連結的一步，防止 AI 誤判直接弄壞正式環境。
3. **告警去重快取 (Alert Storm Mitigation)**：
   整合 DynamoDB 快取與 TTL 機制，同個資源的相同故障在冷卻期內只會通知一次，徹底解決維運人員的「告警疲勞」痛點。
4. **VPC 內網維護機器人 (Isolated Remediation)**：
   自動修復專案部署於 VPC 私有網段，以最安全的內網通道存取 EKS 控制面，兼顧「自動修復便利性」與「網路隔離安全性」。
